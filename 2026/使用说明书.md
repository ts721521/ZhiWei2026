# Office 文档批量转换 & 梳理工具使用说明书

**版本**: v5.16.0  
**更新日期**: 2026-02-10

---

## 1. 软件定位

本工具用于把 Office 文档批量处理为 PDF，并支持合并、梳理去重，以及面向 NotebookLM 的来源反查。

核心价值：

- 批量转换（Word/Excel/PPT -> PDF）
- PDF 合并（分类拆卷 / 全部合一）
- 文件梳理（去重、索引）
- NotebookLM 溯源定位（`合并PDF+页码` 或 `短ID`）
- 本机检索增强（Everything / Listary）

---

## 2. 界面结构（双 Tab）

- `运行中心`（高频）
- `配置中心`（低频默认参数）

### 2.1 运行中心

用于本次任务执行：

- 运行模式：仅转换 / 仅合并 / 转换+合并 / 归集
- 路径：源目录、目标目录
- 执行参数：引擎、沙箱、合并参数、日期过滤
- NotebookLM 快速定位：`刷新Map`、`定位`、`打开文件`、`打开目录`、`Everything 搜索`、`复制 Listary 查询`

### 2.2 配置中心

用于长期默认值：

- 日志目录
- 超时与限制参数
- 排除目录与关键词
- Tooltip 样式（延迟、颜色、字体）
- 保存到 `config.json`

## 3. 数据流程（重点核对）

## 3.1 主流程 A：转换 / 合并

1. 输入：`source_folder` + `target_folder` + 运行参数  
2. 转换阶段（可选）：Office -> PDF  
3. 合并阶段（可选）：生成 `Merged_xxx.pdf`  
4. 输出目录：`<target_folder>/_MERGED/`  
5. 同步输出映射文件：
- `Merged_xxx.map.json`
- `Merged_xxx.map.csv`

映射文件记录每个源文件在合并 PDF 中的页码区间、短ID、路径等元数据。

## 3.2 主流程 B：NotebookLM 溯源定位

1. 你在 NotebookLM 拿到引用（常见是“合并文件 + 页码”）
2. 在 GUI 定位区输入：
- 主路径：`合并文件 + 页码`
- 备路径：`短ID`
3. 点击 `定位`
4. 程序在 `_MERGED` 中读取 `*.map.json`，回查源文件记录
5. 成功后可：
- 打开源文件
- 打开所在目录
- 用 Everything 二次检索
- 复制 Listary 查询语句

## 3.3 按钮灰态规则（你当前最关心）

`Everything 搜索` 和 `复制 Listary 查询` 在以下条件前会是灰色：

- 还没定位成功（没有 `last_locate_record`）

它们会在“定位命中后”自动亮起。

## 4. 增量同步与增量包使用指南（新增）

本功能专为应对海量文件（如 10,000+ 个文档）设计，能显著减少重复转换时间，并生成便于维护的小体积增量包。

### 核心概念：怎么做到“只转新文件”？

程序内部维护了一个 **“电子账本”**（`incremental_registry.json`）。
- **记账**：记录所有已处理文件的“指纹”（大小、修改时间、哈希值）。
- **查账**：下次运行时，程序会先拿当前文件和“账本”比对。
- **决策**：
  - **没变** -> 直接跳过（秒级完成）。
  - **变了**（新增/修改） -> 加入转换队列。
  - **改名了** -> 自动识别为“重命名”，不重新转换，只更新账本路径。

### 4.1 场景一：第一次运行（初始化）

面对 12,000 个文件，第一次必须进行全量转换，建立初始账本。

**操作步骤**：
1. **设置路径**：设置好源目录和目标目录。
2. **开启增量**：
   - 在 **配置中心 -> 增量与去重** 标签页中，勾选：
     - `[x] 启用增量扫描`
     - `[x] 生成增量包`
     - `[x] 同目录同名优先 Office`（推荐：避免重复转换同一份文档的 PDF 版）
     - `[x] 按类型全局 MD5 去重`（推荐：自动跳过内容完全重复的文件）
3. **开始运行**：点击“开始运行”。
   - **耐心等待**：第一次会比较慢（取决于文件量），建议空闲时间挂机运行。
   - **结果**：运行结束后，目标目录下会生成全量 PDF，并在 `_AI/registry/` 下自动生成账本文件。

### 4.2 场景二：日常更新（增量运行）

当您新增了 50 个文件，或修改了 10 个文件后：

**操作步骤**：
1. **保持配置不变**：确保“启用增量扫描”依然勾选。
2. **直接运行**：点击“开始运行”。
3. **极速完成**：
   - 程序会飞快地扫描那 12,000 个旧文件并显示“Skipped（跳过）”。
   - **只转换** 那 60 个变动的文件。
4. **收取增量包**：
   - 这次**不会**重写覆盖原本那 1 万个 PDF。
   - 而是生成一个 **独立增量包**，位置在：`<目标目录>/_AI/Update_Package/Update_Package_2026xxxx_xxxx/`。

### 4.3 增量包里有什么？

增量包是专门给 NotebookLM 或 RAG 系统“投喂”的，它是**纯净的增量数据**：

- **`PDF/` 文件夹**：只包含本次新增/修改的那几个 PDF 文件。
- **`incremental_manifest.json`**：增量清单（本次新增几个、删除几个）。
- **`incremental_index.xlsx`**：详细的文件变动列表（含重命名追踪记录）。

**使用建议**：
- **给 NotebookLM**：直接把 `PDF/` 文件夹里的新文件上传即可。
- **给 RAG 系统**：读取 `manifest.json`，根据指示添加新文件、删除旧文件索引。

## 5. Map 数据契约（定位依据）

每条记录核心字段：

- `merged_pdf_name`
- `source_filename`
- `source_abspath`
- `source_md5`
- `source_short_id`
- `start_page_1based`
- `end_page_1based`
- `page_count`

定位逻辑：

- 按页码定位：命中 `start_page_1based <= page <= end_page_1based`
- 按短ID定位：命中 `source_short_id`

## 6. Everything / Listary 集成说明

## 6.1 Everything 集成

用途：在定位成功后，用本机索引快速检索对应文件。

配置节点（`config.json`）：

```json
"everything": {
  "enabled": true,
  "es_path": "",
  "prefer_path_exact": true,
  "timeout_ms": 1500
}
```

字段解释：

- `enabled`：是否启用 Everything 功能
- `es_path`：`es.exe` 绝对路径，留空则自动探测
- `prefer_path_exact`：是否带目录做精确检索（推荐 `true`）
- `timeout_ms`：命令超时毫秒

`es.exe` 自动探测顺序：

1. `es_path`（手动指定）
2. 系统 PATH（`es.exe` / `es`）
3. 常见目录：
- `C:\Program Files\Everything\es.exe`
- `C:\Program Files (x86)\Everything\es.exe`

诊断命令（Windows）：

```bat
where es.exe
```

## 6.2 Listary 集成

当前为轻集成：

- 点击 `复制 Listary 查询` 只会复制查询语句到剪贴板
- 不会自动控制 Listary 窗口

查询语句包含：`id/md5/file/path` 组合信息，便于你在 Listary 中粘贴后检索。

## 8. 快速核对清单（建议按此顺序）

1. 跑一次“转换+合并”，确认 `_MERGED` 下有：
- `Merged_xxx.pdf`
- `Merged_xxx.map.json`
- `Merged_xxx.map.csv`
2. 在定位区点击 `刷新Map`，确认有“已加载 N 个映射文件”提示
3. 用 `合并文件 + 页码` 做一次定位，确认命中并可打开源文件
4. 定位成功后，确认 `Everything 搜索` 和 `复制 Listary 查询` 不再灰色
5. 若 Everything 失败，先检查 `es.exe`：
- `where es.exe`
- 或在 `everything.es_path` 写死绝对路径

## 9. 常见问题（更新）

**Q1: 为什么 Everything / Listary 按钮是灰色？**  
A: 需要先成功执行一次“定位”。未命中前按钮会保持灰色。

**Q2: 定位输入页码后提示无效？**  
A: 页码必须是正整数，并且是“合并后 PDF 的页码”，不是源文件页码。

**Q3: NotebookLM 没给页码怎么办？**  
A: 用短ID定位（前提是来源文本或你的备注里带短ID）；否则先用文件名/关键词在 Everything 或 Listary 辅助检索。

**Q4: 改了配置不生效？**  
A: 在配置中心点“保存配置”，然后重新运行任务；运行中心参数会覆盖本次执行。

## 12. MSHelp API Docs Mode (NotebookLM)

This mode is independent from Office conversion and is designed for software API help packs installed under paths like:

`...\Program\...\MSHelpViewer`

Use this mode when your source is Microsoft Help CAB packages and you need NotebookLM-friendly outputs.

- Run mode: `MSHelp API Docs`
- Input scan: recursively find `MSHelpViewer` folder name (configurable)
- Source type: `.cab`
- Primary output: Markdown
- Optional output: DOCX / PDF
- Merge packaging: split by max docs + max chars to fit NotebookLM upload limits
- Source traceability: every merged package contains source map entries, and index files are exported:
  - `_AI/MSHelp/MSHelp_Index_*.json`
  - `_AI/MSHelp/MSHelp_Index_*.csv`
  - `_AI/MSHelp/Merged/MSHelp_API_Merged_*.md` (and optional `.docx` / `.pdf`)

Recommended defaults:

- `mshelpviewer_folder_name = "MSHelpViewer"`
- `enable_mshelp_merge_output = true`
- `mshelp_merge_max_docs = 120`
- `mshelp_merge_max_chars = 1200000`
- `enable_mshelp_output_docx = false`
- `enable_mshelp_output_pdf = false`
