# AGENTS 方案：第一性原理与自我批判式审查

## 一、第一性原理：方案的真正目的

**要解决的问题**（不可再约简）：

1. **多人/多 AI 协作时**：谁接手都能在最短路径内知道「项目现状」和「接下来该做什么」。
2. **可追溯**：每次代码变更和每次测试结果都有记录，且与「单一事实来源」一致。
3. **可执行**：规则能被 AI 或人无歧义地执行，且能通过工具或流程部分强制。

因此，方案的本质是：

- 定义**唯一入口**（AGENTS.md）：规定「去哪读现状」「去哪读待办」「改代码时要更新哪几个文档」「跑测试时要记到哪里」。
- 规定**必要更新**：代码变更 → 更新计划/交接文档；跑测试 → 更新测试汇总；结构/路径变更 → 更新 AGENTS 本身。
- 规定**接手顺序**：先读 AGENTS，再按 AGENTS 指向的文档读现状与待办。
- **通用模板**：在无现成结构时，通过「检测 → 沿用或创建 → 填写路径」得到与上面一致的约定。

以下用苏格拉底式追问逐条审查，只保留有依据、可落地、无漏洞的结论。

---

## 二、逐条追问与结论

### 1. 「接手必读」四步顺序

| 追问 | 结论 |
|------|------|
| **依据**：为何必须按顺序？ | 不规定顺序则不同 AI 会读不同东西优先，行为不一致；规定顺序则「先规则、再现状、再待办」可复现。 |
| **可行性**：若文档不存在怎么办？ | 模板已规定：无则执行初始化清单。但「待办」与「交接/现状」若拆成多处，初始化要创建多份；若待办可并入交接文档的「下一步建议」，则依赖更少。 |
| **风险** | 待办单独成 TASK_LIST + plans/ 时，新项目可能两处都空，接手者仍不知「要干什么」。若交接文档必含「下一步建议」一节，则单点即可。 |
| **反面** | 「执行者/会话」对「接手」非必需，对「多 AI 追溯」有用。保留为**建议**即可。 |

**保留**：顺序（读 AGENTS → 读现状/交接 → 读待办）。**精简**：待办可与交接合并为「交接文档含下一步建议」；执行者保持为建议。

---

### 2. 「文档目录：已有则沿用，无则当前项目创建，用户不想则指定」

| 追问 | 结论 |
|------|------|
| **依据** | 已有目录时沿用可减少误改他人结构；无则默认当前项目创建可减少每次询问。 |
| **可行性** | 「当前项目目录」可定义为工作区根或 git 根，可落地。 |
| **风险** | 当前项目只读或无写权限时，自动创建会静默失败。 |
| **反面** | 用户已明确要求「没有就自动创建在当前项目」，不收回；需补漏。 |

**保留**：三支逻辑（沿用 / 当前项目创建 / 用户指定）。**补漏**：若创建失败（如权限），则提示用户指定路径。

---

### 3. 初始化清单：「检测 → 确认是否当前项目创建 → 创建各目录与文件」

| 追问 | 结论 |
|------|------|
| **依据** | 统一清单保证不同 AI 初始化结果一致。 |
| **可行性** | 「等价路径」若写成 doc/plans/、test-reports/ 等开放列举，AI 可能猜错。只检查「本项目已填写的路径」或「默认 docs/plans/、docs/test-reports/」更可验证。 |
| **风险** | 项目用 documentation/、spec/ 等非约定名时，会视为「无」并新建 docs/，可能造成两套目录。接受：模板约定默认名，特殊项目由用户在 AGENTS 中填写实际路径。 |
| **反面** | 无。 |

**保留**：清单结构。**精简**：「等价路径」改为明确列举（如 docs/plans/、docs/test-reports/ 或 AGENTS 中已填路径），去掉「等等价路径」等模糊表述。

---

### 4. 基本原则：「可追溯、文档是交付物、结构迁移更新 AGENTS」

| 追问 | 结论 |
|------|------|
| **依据** | 可追溯是接手与排查的基础；文档为交付物则不会被视为可选项；结构迁移不更新 AGENTS 则路径失效。 |
| **可行性** | 可追溯可通过 git 或文档摘要实现；「结构迁移」需可判定。 |
| **风险** | 「结构迁移」过宽会变成任何改目录都要改 AGENTS。应收窄为「导致 AGENTS 所列路径或目录失效的变更」。 |
| **反面** | 无。 |

**保留**：三条原则。**补漏**：将「任何结构迁移」改为「任何导致本文档所列路径或目录失效的变更」。

---

### 5. 强制文档同步：「代码路径变更 → 更新计划+测试汇总；架构迁移 → 更新 AGENTS」

| 追问 | 结论 |
|------|------|
| **依据** | 这是「代码与文档一致」的核心不变量。 |
| **可行性** | 2026 用 check_doc_sync 脚本可检查；模板中为可选。规则本身不依赖脚本。 |
| **风险** | 若项目未填写「约定的代码路径」，规则无法适用。应规定：未填写则视为「任意代码变更均需更新文档」。 |
| **反面** | 是否必须「同时」更新两处？若仅改注释或仅跑测试，可只更新一处。更精确：有代码变更 → 计划/交接记摘要；有跑测试 → 测试汇总记结果；两者可同一次完成。 |

**保留**：代码变更 → 更新计划/交接；跑测试 → 更新测试汇总；结构/路径失效 → 更新 AGENTS。**补漏**：未填写代码路径约定时，任意代码变更均触发文档更新；允许「仅更新测试汇总」或「仅更新计划/交接」当且仅当本轮仅跑测试或仅改文档。

---

### 6. 强制测试记录：「一轮开发后跑测试并记录命令/用例数/结果/摘要/执行者」

| 追问 | 结论 |
|------|------|
| **依据** | 测试汇总的「最新结果」必须有人写入，否则接手者会看到过期数据。 |
| **可行性** | 「一轮开发后」略模糊，可接受；执行者已标为建议。 |
| **风险** | 项目无自动化测试时，命令可能失败或无意义。 |
| **反面** | 无。 |

**保留**：跑测试并记录。**补漏**：若项目无自动化测试，可在测试汇总注明「无」或「待补充」，并仍在计划/交接中记录本轮摘要。

---

### 7. 禁止项、执行口径、可选校验与 CI

| 条目 | 结论 |
|------|------|
| 禁止项 | 为正规则的逆表述，保留，简洁无赘。 |
| 执行口径 | 冲突时以可追溯优先、例外须注明，保留。 |
| 可选校验与 CI | 明确可选，保留不展开。 |

---

### 8. 模板中的重复与模糊

| 问题 | 结论 |
|------|------|
| 开头「复用方式」与「使用说明」重复同一套逻辑 | 合并为一段，只保留一套：沿用 / 当前项目创建 / 用户指定。 |
| 「本项目的文档目录」与「初始化清单」中「沿用/创建/指定」写两遍 | 目录节只保留路径表格与「填写说明」；创建逻辑只在初始化清单中写一次。 |
| 「等等价路径」 | 删除，改为明确列举或「本项目已填路径」。 |

## 三、保留后的硬核清单（可落地、无漏洞）

- **唯一入口**：AGENTS.md 规定「现状/交接文档路径」「测试汇总路径」「代码路径约定」「测试命令」。
- **接手顺序**：读 AGENTS → 读现状/交接（含下一步建议）→ 可选读单独任务清单。
- **必要更新**：  
  - 有代码变更 → 更新计划/交接（或测试汇总）中的变更摘要；  
  - 有跑测试 → 更新测试汇总；  
  - 导致 AGENTS 所列路径失效的变更 → 更新 AGENTS。  
  - 未填写代码路径约定时，任意代码变更均触发文档更新。
- **初始化**：检测 docs/plans/、docs/test-reports/（或 AGENTS 已填路径）→ 有则沿用；无则默认当前项目下创建，若创建失败则请用户指定路径；创建最小交接文档与测试汇总占位；将实际路径回写 AGENTS。
- **原则**：可追溯、文档为交付物、路径失效变更须更新 AGENTS。
- **例外**：冲突时以可追溯优先；例外须在交接文档中注明原因、范围与后续计划。
- **执行者/会话**：建议记录，不强制。
- **无自动化测试**：允许在测试汇总注明「无」或「待补充」。

---

## 四、对通用模板的修改建议（已用于精简版）

1. 合并开头两段为一段（沿用 / 当前项目创建 / 用户指定 + 创建失败则请用户指定）。
2. 接手必读：待办与「下一步建议」合并表述；执行者保持为建议。
3. 文档目录节：只保留路径表与「优先沿用；无则当前项目创建；不想则指定；创建失败则请用户指定」一句，删除与初始化清单的重复描述。
4. 初始化清单：明确「仅检查 docs/plans/、docs/test-reports/ 或 AGENTS 已填路径」；增加「若创建失败则请用户指定路径」；可选交接总览/任务清单不变。
5. 基本原则：将「任何结构迁移」改为「任何导致本文档所列路径或目录失效的变更」。
6. 强制文档同步：增加「未填写代码路径约定时，任意代码变更均触发文档更新」；允许仅更新测试汇总或仅更新计划/交接（按本轮实际）。
7. 强制测试记录：增加「若项目无自动化测试，则在测试汇总注明『无』或『待补充』，并仍建议在计划/交接中记录本轮摘要」。
8. 删除所有「等等价路径」类模糊表述。

以上为自我批判式审查结论；通用模板的精简版见 `AGENTS_TEMPLATE.md` 的后续修订。
