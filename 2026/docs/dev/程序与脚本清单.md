# 程序与脚本清单

本清单用于说明 `2026/` 仓库中各主要程序与脚本的**职责分工**，帮助读者区分：

- **主程序 / 被测对象**：实际提供功能的应用与库（例如 `OfficeConverter`）。
- **测试 / E2E 驱动**：调用主程序、检查产物的脚本（不重写业务逻辑）。
- **工具 / 诊断脚本**：辅助排查问题、配置环境或同步文档的脚本。

> 约定：若本清单提到的文件路径或职责发生变化，修改对应代码时必须**同步更新本清单**。

---

## 一、主程序（被 E2E 测试的对象）

| 名称 | 路径 | 职责 | 说明 |
|------|------|------|------|
| 知喂 GUI | `office_gui.py` | 图形界面入口。负责渲染界面、读取/保存配置，用户在此配置源目录、目标目录、运行模式、LLM 交付等参数，并点击「开始」执行。 | 内部使用 `OfficeConverter` 和 `converter/**` 中的逻辑完成实际转换与合并。 |
| 转换/合并核心 | `office_converter.py` 中的 `OfficeConverter` 类 | 核心业务入口。负责初始化配置、扫描源文件、调度批量转换、调用合并流程、生成增量包、写入各种导出（包括 `_LLM_UPLOAD`）。 | 所有 CLI/GUI/E2E 场景最终都通过 `OfficeConverter` 完成转换与合并。 |
| 运行工作流 | `converter/run_workflow.py` 中的 `run(converter, ...)` | 顶层运行工作流。按 `run_mode` 执行：扫描候选文件、并行/串行转换、失败重试、合并 PDF/Markdown、生成索引与增量包、调用 `corpus_manifest` 与 LLM Hub 构建。 | `OfficeConverter.run()` 内部委托到该函数。被 E2E 直接调用。 |
| 语料清单与 LLM Hub | `converter/corpus_manifest.py` | 汇总本次运行产生的所有产物（PDF、Markdown、合并件、索引、增量包等），写入 `corpus.json`；根据配置构建 `_LLM_UPLOAD` 目录及 `llm_upload_manifest.json`、`README_UPLOAD_LIST.txt`。 | NotebookLM / LLM 上传相关的集中输出逻辑所在。 |

## 二、NotebookLM 相关 E2E 测试脚本（驱动 + 断言）

这些脚本**不会重写转换/合并逻辑**，而是通过 `OfficeConverter` 调用真实程序，并对产物做自动化检查。

| 名称 | 路径 | 职责 | 说明 |
|------|------|------|------|
| NotebookLM E2E 测试脚本 | `scripts/run_notebooklm_e2e.py` | 无交互地执行一次 NotebookLM 场景的端到端转换与合并，并检查产物。具体步骤：① 生成或加载配置（以默认配置为基，应用 NotebookLM 推荐覆盖项）；② 实例化 `OfficeConverter(config_path, interactive=False)` 并调用 `converter.run()`；③ 读取 log 与 `<target>/_LLM_UPLOAD` 目录，检查是否存在有效的 LLM Hub 与 manifest、统计文件数与单文件大小；④ 写入结果 JSON 与修复提示文件，返回退出码 0/1/2。 | **被测试对象 = `OfficeConverter` + `converter/run_workflow.run()` 等核心模块；本脚本仅负责驱动与断言，不包含任何转换或合并的业务实现。** |
| NotebookLM 全量 E2E 批处理 | `scripts/run_notebooklm_e2e_full.bat` | 在 Windows 终端中运行，用于一次性执行全量 NotebookLM E2E 并将输出重定向到带时间戳的日志文件。实质上是对 `python scripts/run_notebooklm_e2e.py` 的包装。 | 方便长时间运行与归档日志，不改变被测程序。 |

## 三、工具与诊断脚本（非被测业务逻辑）

这些脚本用于**诊断、配置或同步文档**，不会替代主程序逻辑。

| 名称 | 路径 | 职责 | 说明 |
|------|------|------|------|
| 转换范围诊断 | `scripts/diagnose_convert_scope.py` | 加载指定配置并实例化 `OfficeConverter`，扫描「会被本次 run 处理的文件列表」，输出统计信息与示例路径。**不执行真正的转换或合并**。 | 用于在大目录下预估转换范围与耗时。 |
| Google Drive 配置检查 | `scripts/check_gdrive_config.py` | 检查 Google Drive 上传所需的 client secrets、token 等本地文件是否存在、可读，并输出提示。 | 配合 GUI 中的「上传 _LLM_UPLOAD 到 Google Drive」功能使用。 |
| 文档同步校验 | `scripts/check_doc_sync.py` | 根据变更文件列表（命令行显式传入或使用 `--staged` 自动读取 git 暂存区），检查是否同步更新约定的计划文档、测试汇总和 `AGENTS.md`。 | 建议在提交前或 CI 中运行，避免出现「只改代码不记文档」的情况。 |
| Git 钩子安装 | `scripts/install_git_hook.py` | 将仓库提供的 `.githooks/pre-commit` 安装为本地 git pre-commit 钩子，使得在提交前自动执行 `check_doc_sync.py` 等检查。 | 可选，但推荐在本地开发环境中启用。 |
| 文档同步到 Obsidian | `scripts/sync_docs_to_obsidian.py` | 将选定的项目文档同步到本地 Obsidian 笔记库中，便于知识管理与离线浏览。 | 不参与运行时逻辑。 |
| MarkItDown 冒烟脚本 | `scripts/markitdown_smoke_mvp.py` | 对 MarkItDown 相关能力做简单冒烟测试，验证依赖是否可用。 | 单独的小工具，不影响主流水线。 |

## 四、如何使用本清单

- **开发/接手前**：先阅读本清单，确认自己要改动的是「主程序」「E2E 驱动」还是「工具脚本」，避免误把 E2E 当业务实现、或在工具脚本里重复造轮子。
- **编写/调整 E2E 时**：确保 E2E **只调用主程序并做断言**，不在脚本中直接实现业务逻辑；若有新增脚本，请在本清单中登记其职责与被测对象关系。
- **代码评审时**：可对照本清单快速判断某次改动是否影响主程序行为、是否需要新增或更新对应的测试与文档。
